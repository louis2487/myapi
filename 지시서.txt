목표:
기존 서버 A(api.smartgauge.co.kr)의 volume(/data) 안에 있는 이미지 파일들을 새 서버 B(api.daewon469.com)의 volume(/data) 안으로 마이그레이션하는 1회성 기능을 FastAPI에 추가하라.

서버 정보:
- A 서버 도메인: https://api.smartgauge.co.kr
- A 서버 volume mount path: /data
- B 서버 도메인: https://api.daewon469.com
- B 서버 volume mount path: /data

가정:
- A 서버에는 실제 이미지 파일이 /data/uploads 아래 저장되어 있다.
- A 서버는 외부에서 /uploads/{filename} 형태로 이미지 접근이 가능하거나, 가능하도록 만들 수 있다.
- B 서버에도 이미지 저장 경로는 /data/uploads 로 맞춘다.
- 이번 기능은 "1회성 마이그레이션용 임시 API" 이며, 작업 완료 후 제거 가능하도록 작성한다.

해야 할 일:
1. A 서버에 업로드 파일 목록을 반환하는 임시 API 추가
2. B 서버에 A 서버로부터 파일 목록을 받아 순차 다운로드 후 /data/uploads 에 저장하는 임시 API 추가
3. 이미 존재하는 파일은 기본적으로 skip 처리
4. 성공/실패/skip 개수를 JSON으로 반환
5. 코드가 너무 복잡해지지 않게, 유지보수 가능한 수준으로 분리
6. 작업 완료 후 쉽게 제거할 수 있도록 migration 전용 라우터/서비스 파일로 분리

구현 상세 요구사항:

[A 서버 구현]
- GET API 1개를 만든다.
- 예시 경로:
  GET /admin/migration/uploads/files
- 기능:
  - /data/uploads 디렉토리를 재귀적으로 탐색한다.
  - 파일 목록을 relative path 기준으로 반환한다.
  - 예:
    {
      "base_url": "https://api.smartgauge.co.kr/uploads",
      "count": 3,
      "files": [
        "img_1.jpg",
        "2026/03/img_2.png",
        "nested/sample.webp"
      ]
    }
- 응답에 디렉토리는 포함하지 말고 실제 파일만 포함한다.
- 정렬은 안정적으로 되도록 sorted 처리한다.

[B 서버 구현]
- POST API 1개를 만든다.
- 예시 경로:
  POST /admin/migration/uploads/pull-from-smartgauge
- 기능:
  1) A 서버의 파일 목록 API를 호출한다.
  2) 반환된 files 배열을 순회한다.
  3) 각 파일에 대해 A 서버의 /uploads/{relative_path} URL로 다운로드 요청을 보낸다.
  4) B 서버의 /data/uploads/{relative_path} 에 저장한다.
  5) 중간 디렉토리가 없으면 mkdir(parents=True, exist_ok=True) 로 생성한다.
  6) 이미 동일 경로의 파일이 있으면 기본 동작은 skip 한다.
  7) 옵션으로 overwrite=true 가 들어오면 덮어쓴다.
- 응답 형식 예시:
  {
    "success": true,
    "source": "https://api.smartgauge.co.kr",
    "target": "https://api.daewon469.com",
    "total": 120,
    "downloaded": 110,
    "skipped": 8,
    "failed": 2,
    "failed_files": [
      {
        "path": "broken/a.jpg",
        "reason": "404 Not Found"
      },
      {
        "path": "timeout/b.png",
        "reason": "timeout"
      }
    ]
  }

추가 요구사항:
- B 서버 쪽 다운로드는 한 번에 과도한 병렬처리 하지 말고, 우선 순차 처리로 구현한다.
- timeout을 명시한다.
- requests 대신 현재 프로젝트 스타일에 맞는 HTTP 클라이언트를 사용하되, 특별한 이유가 없으면 httpx 사용.
- 파일 저장은 바이너리 모드로 안전하게 처리한다.
- path traversal 방지 로직을 넣는다.
  - 예: ../../ 같은 경로가 들어오면 저장하지 않도록 검증
  - 반드시 /data/uploads 밖으로 벗어나지 못하게 한다.
- content-type에 너무 의존하지 말고, 응답 status code 와 body 기준으로 저장한다.
- 큰 파일도 처리 가능하도록 stream download 방식으로 구현한다.
- 각 파일마다 예외를 잡아 전체 작업이 중단되지 않게 한다.
- 로그를 남기되 과도한 출력은 피한다.

환경변수:
- B 서버:
  - SMARTGAUGE_SOURCE_BASE_URL=https://api.smartgauge.co.kr
  - SMARTGAUGE_SOURCE_FILES_ENDPOINT=/admin/migration/uploads/files
  - SMARTGAUGE_SOURCE_UPLOADS_BASE_URL=https://api.smartgauge.co.kr/uploads

경로 규칙:
- A 실제 파일 루트: /data/uploads
- B 실제 파일 루트: /data/uploads
- A 외부 공개 URL 루트: https://api.smartgauge.co.kr/uploads
- B는 저장만 하면 되고, 기존 정적 파일 서빙 방식에 따라 /uploads 로 접근되게 유지

원하는 파일 구조 예시:
- app/api/admin_migration.py
- app/services/upload_migration_service.py
- 필요한 경우 settings/config 파일에 환경변수 추가

구현 포인트:
- list_upload_files(root_dir: Path) -> list[str]
- validate_relative_upload_path(path_str: str) -> str
- build_source_file_url(relative_path: str) -> str
- save_stream_to_target(target_path: Path, response_stream) -> None
- migrate_uploads_from_source(overwrite: bool = False, limit: int | None = None) -> dict

추가 옵션:
- limit 파라미터를 넣어 테스트 가능하게 하라.
  - 예: 처음엔 5개만 받아보는 식
- dry_run=true 옵션도 지원하면 좋다.
  - 실제 저장 없이 몇 개 대상인지, 어떤 파일이 처리될지만 확인 가능하게

체크리스트:
1. A 서버 파일 목록 API 동작 확인
2. B 서버에서 limit=3, dry_run=true 로 테스트
3. B 서버에서 limit=3 실제 저장 테스트
4. /data/uploads 에 파일 생성 확인
5. B 도메인에서 이미지 접근 확인
6. 전체 마이그레이션 실행
7. 실패 파일 재시도 가능하도록 결과 확인
8. 완료 후 임시 API 제거 가능하도록 정리

중요:
- 기존 업로드 로직은 건드리지 말고, migration 전용 코드만 추가하라.
- 기존 정적 파일 서빙 로직도 최대한 변경하지 말라.
- 이번 작업은 운영 데이터 이관이므로, 안정성과 경로 검증을 우선하라.
- 구현 후에는 "추가된 파일 목록", "등록된 엔드포인트", "실행 방법", "테스트 방법"까지 함께 정리하라.
